# Technical Spec: AI 广告推荐信息流 Android App (Tictok Advert)

## 1. 核心技术难点与思考

### 1.1 单列信息流的混排渲染与滚动稳定性

**难点**：首页同时存在大图、小图、视频三种卡片，且需要在同一个信息流里稳定混排。同时还要支持频道切换、下拉刷新和标签过滤，不能让列表状态互相串位。

**思考**：
- 采用 `HorizontalPager + LazyColumn` 作为首页主容器，Pager 负责频道切换，列表负责单频道渲染。
- 用 `ad.id` 作为稳定 key，避免频道切换或刷新时列表项抖动。
- 按频道保存 `LazyListState`，保证 Tab 切换后能回到原位置。
- 刷新只作用于当前频道，并在刷新完成后主动滚回列表顶部，确保“换一批”结果立刻可见。
- 把封面/视频的本地与远程路径收敛到 `AdEntity`，让页面只处理展示，不重复判断。

### 1.2 视频卡片的播放器选型与续播

**难点**：视频卡片既出现在首页 Feed，也出现在详情页。需要兼顾资源占用、播放稳定性和返回后续播。

**思考**：
1. 首页只允许单个视频处于播放态，通过 `playingAdId` 控制同一时间只有一个 `VideoView` 活跃。
2. 详情页用 `VideoView + seekTo()` 续播，而不是引入更重的播放器框架。
3. 进度保存在 `AdvertViewModel` 的 `_videoProgressMap(adId -> positionMs)` 中，离开页面时回写，进入详情页时恢复。

这种方案的目标不是追求最强播放器能力，而是用最小依赖完成可演示、可维护的续播体验。

### 1.3 AI 搜索的结构化输出与降级

**难点**：自然语言搜索如果输出不稳定，后续频道过滤和标签过滤就无法可靠执行；同时云端 API 也可能因 Key 缺失或网络问题失败。

**思考**：
- ModelScope 上的 `Qwen/Qwen3-32B` 只负责把 query 转成结构化 `SearchIntent`。
- 严格限制输出字段为 `channel / tags / explanation`，并要求 JSON 格式。
- 通过 `extra_body.enable_thinking=false` 关闭思考输出，降低延迟并减少额外文本干扰。
- API 失败时立即切换到本地规则解析，保证搜索页仍可用。
- 当前不单独做搜索结果缓存，优先保证输出确定性和离线可用性。

### 1.4 离线广告包的版本化导入

**难点**：素材既要能直接跟仓库交付，又要避免“编译成功但新数据没加载”的问题。

**思考**：
- 把广告数据做成 `manifest.json + 图片/视频资源` 的离线包。
- 使用 `generatedAt` 作为版本号。
- 启动时对比本地已导入版本，版本变更则重导入并覆盖旧资源。
- 数据最终落到 Room，确保页面、统计和搜索消费的是同一份实体数据。

### 1.5 详情页评论的最小持久化闭环

**难点**：课程验收需要看到详情页互动，但项目没有账号体系和服务端评论接口。如果直接把评论放在页面内存状态里，重新进入详情页后数据就会消失。

**思考**：
- 在现有 `Room` 数据库里新增 `comments` 表，避免再引入新的数据源。
- 评论只绑定 `adId`，字段保持最小闭环：`authorName / content / createdAt`。
- 作者名固定为 `我`，不额外扩展登录、昵称和回复链路。
- 通过 `Flow<List<CommentEntity>>` 直接驱动详情页评论区，提交后立即刷新展示。

## 2. 方案对比与最终定型

### 2.1 主信息流 UI 方案对比

| 方案 | 说明 | 优点 | 缺点 |
| --- | --- | --- | --- |
| A. RecyclerView + XML | 传统原生列表 + 多 ViewHolder | 生态成熟，复用机制明确 | 多卡片状态切换和媒体路径分支较重，页面逻辑容易分散 |
| B. Compose `HorizontalPager + LazyColumn` | 声明式频道 Pager + 单频道列表 + 分频道状态保存 | 写法统一，频道滑动和列表恢复更自然 | 需要适配 Compose 的状态同步 |
| **最终选择：B** |  | **更适合当前项目的单列信息流、频道滑动和快速迭代** |  |

**结论**：选择 Compose `HorizontalPager + LazyColumn`。当前项目的核心不是复杂的多层嵌套列表，而是“卡片类型多、频道状态要稳、刷新后要立即呈现新内容”，Compose 的声明式写法更合适。

### 2.2 视频播放方案对比

| 方案 | 说明 | 优点 | 缺点 |
| --- | --- | --- | --- |
| A. ExoPlayer / Media3 | 专业播放器框架 | 能力强、可扩展性好 | 集成成本高，当前项目需要的只是基础播放与续播 |
| B. `VideoView` + Compose Overlay | 系统原生播放器，配合页面状态控制 | 依赖轻、实现快、足够完成 Feed/详情页播放 | 能力不如 ExoPlayer 丰富 |
| **最终选择：B** |  | **能以最小复杂度完成可演示的视频卡片** |  |

**结论**：选择 `VideoView`。当前项目的视频需求偏展示型，不追求复杂 ABR、自定义缓存或高级控制，`VideoView` 更轻、更直观，也更容易和 Compose 页面状态结合。

### 2.3 广告数据方案对比

| 方案 | 说明 | 优点 | 缺点 |
| --- | --- | --- | --- |
| A. 运行时联网拉取 | 页面启动时实时抓取/请求广告 | 数据更新灵活 | 复现性差，答辩现场不稳定，易受网络和页面变化影响 |
| B. 纯 mock 静态数据 | 直接写死一份本地 JSON | 最快 | 无法满足“中文广告素材落地”的要求，也容易继续沿用旧样例 |
| C. 离线包 + Room + 版本号 | 资源随仓库交付，启动导入到本地数据库 | 可复现、可离线、易维护 | 需要维护 manifest 和本地资源目录 |
| **最终选择：C** |  | **最符合结题交付和现场演示要求** |  |

**结论**：采用离线包方案。它把素材、文案、视频都固定在仓库中，保证“编译能过、运行一致、答辩可复现”。

### 2.4 AI 搜索方案对比

| 方案 | 说明 | 优点 | 缺点 |
| --- | --- | --- | --- |
| A. 纯云端 LLM 搜索 | 完全依赖 Qwen3-32B 解析 query | 结果智能、表达自然 | 依赖网络和 Key，失败即不可用 |
| B. 纯本地规则解析 | 只靠关键词和规则匹配 | 离线可用、稳定 | 语义理解能力有限 |
| C. 云端优先 + 本地 fallback | Qwen 优先，失败后回退规则解析 | 兼顾智能与稳定 | 需要同时维护两套逻辑 |
| **最终选择：C** |  | **最适合课程场景下的可交付搜索体验** |  |

**结论**：选择“云端优先 + 本地 fallback”。这样既能展示 AI 能力，又不会让搜索成为单点故障。

### 2.5 评论持久化方案对比

| 方案 | 说明 | 优点 | 缺点 |
| --- | --- | --- | --- |
| A. 页面内存状态 | 评论只存在详情页 `remember` 状态 | 实现最快 | 离开页面即丢失，无法形成闭环 |
| B. SharedPreferences / 本地文件 | 手工序列化评论到本地 | 实现门槛低 | 结构扩展性差，不适合按广告查询 |
| C. Room `comments` 表 | 评论按 `adId` 持久化，Flow 驱动 UI | 与现有数据库一致，易查询，重进页面可恢复 | 需要升级数据库版本 |
| **最终选择：C** |  | **最符合当前项目已有数据层结构** |  |

**结论**：评论直接进入 Room。这样不需要新增服务端，也能在验收场景里展示“输入评论 -> 退出详情 -> 再次进入仍存在”的完整闭环。

## 3. 最终方案与原因

### 3.1 架构

```text
MainActivity
  └─ TictokAdvertApp
      ├─ HomeScreen
      ├─ DetailsScreen
      ├─ SearchScreen
      └─ StatsScreen
            ↓
      AdvertViewModel
        ├─ Room flows / analytics / comments
        ├─ video progress map
        └─ Qwen search
            ↓
      TictokAdvertRepository
        ├─ OfflineAdBundleLoader
        ├─ AdDao / AppDatabase
        └─ ads / analytics_events / comments
```

### 3.2 关键落地

1. 首页使用 `LazyColumn` + 稳定 key 渲染混排广告。
2. 首页使用 `HorizontalPager` 保持 `精选 -> 电商 -> 本地` 的视觉顺序，支持点击 Tab 和左右滑动切换。
3. 首页下拉刷新只重排当前频道内容，并在刷新完成后滚回顶部。
4. 标签栏固定为单行，展示 `全部 + 最多 3 个热门标签 + 更多下拉`，溢出标签使用三列菜单。
5. 视频卡片使用 `VideoView`，首页只允许一个活跃播放实例。
6. 详情页通过 `seekTo()` 恢复视频进度。
7. 详情页评论写入 Room `comments` 表，本地持久化保存。
8. 离线广告包按 `generatedAt` 版本更新。
9. 搜索结果由 Qwen3-32B 输出结构化 JSON，失败时用本地规则兜底。
10. 曝光、点击、互动统一写入 `analytics_events`，统计页直接聚合。

## 4. 效果评估

### 4.1 功能效果

- 单列信息流：已完成
- 频道切换：已完成
- 左右滑动切频道：已完成
- 下拉刷新并回顶：已完成
- 单行标签栏 + 更多下拉：已完成
- 详情页与返回保持位置：已完成
- 详情页评论：已完成
- 曝光 / 点击 / 互动统计：已完成
- AI 搜索：已完成
- 离线中文广告包：已完成

### 4.2 数据效果

- 广告总数：100
- 频道分布：精选 41 / 电商 36 / 本地 23
- 卡片分布：小图 48 / 大图 47 / 视频 5
- 素材来源：100% 今日头条搜索中文素材

### 4.3 性能效果

基于本地文件系统测量：

- 解析 `manifest.json` 约 `0.27 ms`
- 首次复制 105 个素材文件约 `1768.80 ms`

结论：首次导入的成本主要在素材复制，后续启动只做版本校验，开销很低。

### 4.4 稳定性效果

- 无 ModelScope API Key 时，搜索仍可运行。
- 离线包更新后会自动重导入。
- 评论在本地数据库中可复用，重新进入详情页仍可看到。
- 媒体路径逻辑已统一收敛，减少重复分支。

### 4.5 已知限制

- 当前曝光口径不是像素级可视曝光。
- 当前没有独立的搜索结果缓存层。
- 当前接入的是云端 `Qwen3-32B` 搜索能力，不是端侧本地部署模型；当无网或接口异常时，仍依赖本地规则解析兜底。
- 当前下拉刷新是“本地换一批”语义，不是服务端增量拉取。
